# グローバルルール
# 言語設定（最優先）
- 全ての出力・計画・タスク・説明を日本語で記述すること
- Implementation Plan、Task一覧、Step説明も全て日本語
- 英語での出力は一切禁止

## タスク・計画の出力形式
- タスク名: 日本語
- 実装計画: 日本語
- ステップ説明: 日本語
- コードコメント: 日本語可


## 基本動作

- 計画→承認→実装の順で進める。複雑な変更は必ず計画を提示してから着手する
- 不明点・曖昧点は実装前に質問する。推測で進めない
- 既存コードの関連箇所を理解してから変更する
- 確信が持てない事実はWeb検索で確認し、不確実性を明示する
- 2つ以上のアプローチが存在するときはトレードオフを提示する
- ユーザーの表面的な要求の背後にある本当の目的を読み取る

## コーディング規約

- TypeScriptでは型定義必須。`any`禁止。`unknown` + 型ガードを使う
- エラーハンドリングを省略しない。全ての失敗パスを処理する
- マジックナンバー・マジックストリング禁止。定数化またはEnum化する
- 1関数100行を超えたら分割を検討する（単一責任原則）
- 同じロジックが3箇所以上あれば共通化する（DRY原則）
- 早期リターンで深いネストを回避する（ガード節パターン）
- 関数名・変数名は意図が読み取れる命名にする。略語より明確さを優先
- コメントは「なぜ」を書く。「何を」はコードが語る

## テスト

- TDDサイクルを守る: スタブ → 失敗テスト → 実装 → グリーン → リファクタ
- 自明なassertは書かない。実際の欠陥を検出できるテストだけ書く
- 外部サービス（API, DB, ファイルシステム）はモックする
- テスト説明文は `expect` と一致する言葉で書く

## 実装前チェック

- 同じ問題の既存解決策がないか検索してから実装する
- 複雑な機能は設計方針をまず提示し、OKをもらってから実装する
- 依存ライブラリ追加前に既存で代替できないか確認する

## 完了時チェック

- 変更後は必ずビルド・テスト実行し、エラーゼロを確認してから報告する
- 「実装しました」と報告する前に実際に動作確認する
- 意図しないファイルへの変更がないか差分確認する

## セキュリティ

- APIキー・パスワード・シークレットは `.env` のみ。ハードコード絶対禁止
- `.env` は `.gitignore` に必ず追加。コミット前にチェックする
- ユーザー入力は全て検証する（フロント + バック両方）
- 破壊的操作（`rm -rf`、`DROP TABLE`、一括削除）は実行前に必ず確認を求める
- 新規npmパッケージ追加時はダウンロード数・メンテナンス状況を確認する

## Git

- Conventional Commitsを厳守: `feat:` / `fix:` / `docs:` / `refactor:` / `test:` / `chore:` / `perf:`
- 大きな変更は `agent/<feature-name>` ブランチを切る
- 1コミット = 1つの論理的変更
- コミットメッセージは日本語可。「何を変えたか」より「なぜ変えたか」を書く

## UI/UX

- レスポンシブ対応: 375 / 768 / 1024 / 1440px の4ブレークポイント
- WCAG AA準拠（コントラスト比4.5:1以上、キーボード操作対応）
- ローディング・エラー・空状態の3パターンを必ず実装する
- フォームはバリデーション即時フィードバック + 送信時二重確認

## エラー対応

- 根本原因分析を先に行う。表面的な症状で即修正しない
- エラーは隠さず、代替案とともに報告する
- 「分からない」は明示する。推測で進まない
- 同じエラーを2回踏んだらKnowledgeItemsに記録する

## コンテキスト管理

- 長期セッションでは中間成果物をファイルに書き出してコンテキストを節約する
- 大量データは要約してからコンテキストに入れる
- 複数ステップのタスクでは各ステップ後に進捗を `docs/progress.md` に記録する

## KnowledgeItems（知識ベース）

- セッション終了前に新しい知見を該当ファイルに書き込む
- 同じ問題を2回調べたら必ず記録する（3回目を防ぐ）
- 失敗したアプローチも記録する（同じ失敗を繰り返さないため）
- 保存先: 技術的決定→`docs/decisions.md` / 環境設定→`docs/setup.md` / アーキテクチャ→`docs/architecture.md` / パターン→`docs/patterns.md` / 外部API→`docs/api-notes.md` / ユーザー好み→`MEMORY.md`

## セッション開始時

- `AGENTS.md`（または `CLAUDE.md`）があれば読む
- `docs/architecture.md` があれば読む
- `docs/decisions.md` があれば読む
- `MEMORY.md` があれば過去の知見を確認する

## 禁止事項

- .envファイルをgitにコミットしてはならない
- テストなしで「動きます」と報告してはならない
- any型を多用してはならない
- マジックナンバーを放置してはならない
- 破壊的コマンドを確認なしで実行してはならない
- 知見をKnowledgeItemsに保存せずにセッション終了してはならない
- エラーを隠して「完了しました」と報告してはならない
